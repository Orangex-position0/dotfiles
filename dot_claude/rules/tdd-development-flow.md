# AI Agent 行为准则：TDD 测试驱动开发（全局流程规范）

## 1. 总则

### 1.1 概述

为了确保 AI Agent 遵循 TDD（测试驱动开发）的开发模式，Agent 必须严格按照 **Red-Green-Refactor** 三个阶段的循环进行开发。在执行每个阶段前，Agent 必须向开发者明确声明其当前所处的阶段。

本准则旨在确保 Agent 遵循正确的 TDD 开发流程，避免跳过关键步骤。

### 1.2 路径约定

不同语言的「实现代码」和「测试代码」路径不同，本规范中出现的「实现代码」/「测试代码」均为抽象表述：

| 语言 | 实现代码 | 测试代码 |
|------|---------|---------|
| Rust | `src/**/*.rs` | `src/**/*.rs` 中的 `#[cfg(test)] mod tests` + `tests/**/*.rs` |
| Java / Kotlin | `src/main/**` | `src/test/**` |
| Python | 包内模块 | `tests/**` 或包内 `test_*.py` |
| Go | 包内 `*.go` | 同目录 `*_test.go` |

## 2. TDD 三阶段循环

### 2.1 第一阶段：RED（写失败的测试）

#### 2.1.1 目标

编写一个**必然失败**的测试用例，明确定义即将实现的功能需求。

#### 2.1.2 核心准则

- **允许**：Agent 可以创建新的测试文件或添加新的测试方法
- **要求**：
  - 测试必须是失败的（因为对应的实现代码尚未存在或不完整）
  - 一次只测试一个功能点
  - 测试代码要简单清晰
  - 测试名称要明确表达测试意图
- **禁止**：Agent **不能**修改现有实现代码
- **验证**：运行测试必须显示红色（失败状态）

#### 2.1.3 交互示例

- **开发者提示**：「我需要实现一个计算器的加法功能」
- **Agent 回应**：「已激活 **RED 阶段**。我将先编写一个失败的测试用例来定义加法功能的需求。」

### 2.2 第二阶段：GREEN（让测试通过的最简实现）

#### 2.2.1 目标

编写**最简单**的实现代码，让当前失败的测试通过。

#### 2.2.2 核心准则

- **允许**：Agent 可以创建、修改实现代码
- **要求**：
  - 优先考虑最简单的实现方式
  - 专注于满足当前测试用例
  - 快速实现功能让测试通过
- **禁止**：
  - **不能**修改测试代码
  - **不考虑**代码质量和性能优化
  - **不进行**过度设计
- **验证**：运行测试必须显示绿色（通过状态）

#### 2.2.3 交互示例

- **Agent 回应**：「已激活 **GREEN 阶段**。我将实现最简单的代码来让刚才的测试通过，不考虑优化和设计。」

### 2.3 第三阶段：REFACTOR（重构优化）

#### 2.3.1 目标

在保持测试通过的前提下，改进代码的设计、质量和可维护性。

#### 2.3.2 核心准则

- **允许**：Agent 可以重构实现代码
- **要求**：
  - 改进代码设计和质量
  - 消除重复代码
  - 提高代码可读性和可维护性
  - 每次重构后必须运行测试确保通过
- **禁止**：
  - **不能**修改测试的行为和期望
  - **不能**破坏现有功能
- **验证**：重构过程中和完成后，所有测试必须保持绿色

#### 2.3.3 交互示例

- **Agent 回应**：「已激活 **REFACTOR 阶段**。我将重构代码以提高质量，同时确保所有测试保持通过状态。」

## 3. 按场景选择测试策略

不同复杂度的需求适用不同的测试驱动方式：

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| 简单功能（单个方法、逻辑直观） | 先实现，后验证 | TDD 反而增加无谓成本 |
| 复杂业务逻辑（多分支、算法、状态转换） | TDD（先验证，后实现） | 测试用例是最好的需求文档 |
| 存量代码修改 | 安全网保护（见 §4） | 先确保旧逻辑被测试覆盖 |
| 提示词难以描述需求 | TDD | 让代码直接表达需求 |

## 4. 存量代码安全网策略

### 4.1 背景

AI 对增量开发通常效果很好，但对大型存量系统存在天然局限：

- 上下文窗口有限
- 无法完整理解系统历史
- 难以掌握隐式业务规则
- 无法识别「哪些逻辑绝对不能修改」

### 4.2 流程

1. **在 AI Coding 前**：确保旧有逻辑处于单元测试的完全覆盖中
2. **基线校验**：先确保旧有逻辑能通过所有单测
3. **AI 修改**：让 AI 在测试保护下进行修改
4. **回归校验**：若 AI 修改后的代码无法通过所有单测，立即人工查 bug

### 4.3 优点

- **快速验证**：无需人工对比，只需运行单测即可获得即时反馈
- **清晰界定修改边界**：单测保证 AI 修改的代码不会破坏原有功能

## 5. 防 AI 作弊

### 5.1 必要性

AI 可能为通过测试而走捷径（如修改期望值、删除失败测试、吞掉异常）。必须设定明确的 Rule 防止此类行为。

### 5.2 通用规则

- **禁止修改期望值让测试通过**——发现期望值「不合理」时，必须先与开发者确认是测试错还是实现错，禁止单方面修改期望值
- **禁止删除失败测试**——发现测试失败时，先定位根因
- **禁止吞异常 / 禁用断言**——不允许通过 `try-catch 吞异常` / `@Ignore` / `#[ignore]` 等方式让失败测试「变绿」
- **复杂场景必须手写**——AI 生成的测试覆盖常规路径，但极端边界条件、并发竞争、特定时区切换等场景仍需开发者手写

### 5.3 开发者的角色转变

在 AI + TDD 模式下，开发者从「写提示词」变为「写测试场景」：

- 主动设计需求、定义预期结果
- Code Review 时重点检查 AI 是否作弊
- 复杂场景仍需开发者亲自编写测试用例

## 6. TDD 最佳实践

### 6.1 循环节奏

- **小步快走**：每个 Red-Green-Refactor 循环应该很短（几分钟到十几分钟）
- **频繁验证**：每个阶段完成后都要运行测试验证
- **逐步推进**：一次只关注一个小功能点

### 6.2 测试质量要求

- **快速执行**：单元测试应该在秒级内完成
- **独立性**：测试之间不应该有依赖关系
- **可重复性**：测试结果应该是确定的和可重复的
- **清晰命名**：测试方法名应明确表达测试意图

### 6.3 代码质量保证

- **持续重构**：在每个循环的 REFACTOR 阶段改进代码
- **消除重复**：遵循 DRY（Don't Repeat Yourself）原则
- **保持简洁**：代码应该简洁明了，易于理解

### 6.4 流程控制

Agent 在每个阶段转换时，必须：

1. 明确声明即将进入的阶段
2. 说明当前阶段的具体目标
3. 完成阶段后验证结果
4. 确认是否继续下一个循环
