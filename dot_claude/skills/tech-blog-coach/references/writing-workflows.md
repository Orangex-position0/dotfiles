# 技术文章写作工作流程

基于费曼学习法的工作流程，帮助你将"掌握的知识"转化为"别人能看懂的文章"。

---

## 核心理念：费曼学习法

**目标**：将"你掌握的知识"转为"别人能看懂的文章"

**核心原则**：如果你不能简单地说清楚，说明你还没有完全理解它。

**笔记 vs 文章**：
- **笔记**：给自己看的，包含搜索的资料、碎片信息、长期积累
- **文章**：给读者看的，用费曼学习法检验自己是否真懂

---

## 两种创作模式

### 模式1：基于笔记创作（推荐）

**适用场景**：你已经通过搜索资料整理了笔记，希望将笔记转化为文章

**特点**：
- AI 帮助将笔记映射到5部分结构
- 补充过渡和上下文
- 确保文章面向读者而非自己

**流程**：
```
准备笔记 → 分析笔记 → 映射结构 → 转化补充 → 优化验证
```

### 模式2：自主创作

**适用场景**：你有明确的主题，希望 AI 提供大纲和指导

**特点**：
- AI 提供结构指导
- 你自己对照笔记写作
- AI 提供反馈和优化建议

**流程**：
```
确定主题 → 确定读者 → 结构设计 → 编写初稿 → 优化
```

---

## 模式1：基于笔记创作（详细流程）

### 第0步：准备笔记

**笔记应该包含**：
- 主题背景和问题
- 核心概念和原理
- 代码示例
- 参考资料链接
- 实践中的经验和踩坑

**笔记格式建议**：
- 使用 Markdown 格式
- 有清晰的标题结构
- 代码块标注语言类型
- 重要的概念有说明

---

### 第1步：分析笔记内容

**识别核心要素**：

1. **主题是什么？**
   - 笔记讨论的核心技术或问题
   - 提炼为一句话："使用 xxx 处理 xxx"

2. **目标读者是谁？**
   - 笔记中记录了哪些基础知识？
   - 这些知识适合什么水平的读者？

3. **核心价值是什么？**
   - 笔记解决了什么问题？
   - 读者能从中学到什么？

**示例分析**：
```
笔记主题：Rust 所有权系统
→ 一句话：使用 Rust 所有权系统处理内存安全问题
→ 目标读者：有其他编程语言基础，但不熟悉 Rust 的开发者
→ 核心价值：了解如何在编译时保证内存安全
```

---

### 第2步：映射到5部分结构

将笔记内容分配到5个部分：

#### 1. 背景 & 问题

**从笔记中提取**：
- 问题背景：为什么这个主题重要？
- 实际场景：什么情况下会遇到这个问题？
- 学习价值：读者能获得什么？

**笔记中寻找**：
- 问题描述或痛点
- 技术的动机或背景
- 你最初学习时的困惑

**转化建议**：
- 用故事或场景开头
- 说明为什么这个问题值得解决
- 明确读者能学到什么

---

#### 2. 方案或原理

**从笔记中提取**：
- 核心概念列表
- 技术方案的原理
- 关键术语的解释

**笔记中寻找**：
- 概念定义和说明
- 官方文档的摘录
- 你对原理的理解

**转化建议**：
- 将概念列表转化为结构化的讲解
- 用类比帮助理解抽象概念
- 补充代码框架或架构图
- 分点讲述，每点一个概念

---

#### 3. 实现步骤

**从笔记中提取**：
- 代码示例
- 实现过程
- 关键步骤

**笔记中寻找**：
- 你记录的代码片段
- 实践中的步骤
- 遇到的问题和解决

**转化建议**：
- 将代码片段组织成完整的可运行示例
- 按逻辑顺序排列步骤
- 每个步骤先说明目标，再展示代码
- 确保代码在 IDE 中验证过

---

#### 4. 示例 / 踩坑

**从笔记中提取**：
- 完整的示例代码
- 遇到的错误
- 边界情况
- 调试技巧

**笔记中寻找**：
- 你踩过的坑
- 错误信息和解决方案
- 特殊场景的处理
- 实用的小技巧

**转化建议**：
- 提供一个完整的端到端示例
- 列出常见的错误和原因
- 说明如何调试和排查
- 分享你从实践中获得的经验

---

#### 5. 总结

**从笔记中提取**：
- 核心要点
- 适用场景
- 相关资源

**笔记中寻找**：
- 你记录的关键收获
- 参考资料的链接
- 进一步学习的方向

**转化建议**：
- 总结3-5个核心要点
- 明确说明适用场景
- 诚实承认局限性
- 提供延伸阅读资源

---

### 第3步：转化和补充

**补充内容**：
- 过渡句和连接词
- 背景信息和上下文
- 类比和比喻
- 完整的代码示例

**润色语言**：
- 将速记转化为正式表达
- 补充专业术语的首次解释
- 调整为面向读者的语气
- 确保内容连贯不跳步

**笔记 vs 文章的语言差异**：

```
笔记（给自己）：
"Redis快，因为内存"

文章（给读者）：
"Redis 将数据存储在内存中，因此读写速度远快于基于磁盘的数据库。
一般来说，内存访问速度是磁盘的10万倍以上。"
```

---

### 第4步：优化验证

使用 `assets/templates/writing-checklist.md` 检查：

**核心检查问题**：
- 是否跳过太多步骤？（内容要连贯）
- 是否假设了过多前置知识？（必要概念需补充）
- 代码是否能运行？（先在 IDE 中验证）

## 模式2：自主创作（详细流程）

### 第1步：确定主题

**从日常开发中寻找主题**：

**最佳主题来源**：
1. 刚解决的 Bug
2. 刚学会的框架 / 设计模式
3. 性能优化经历
4. 踩坑经历

**将主题提炼为一句话**：

**固定句式**：
```
使用 [xxx]（技术方案） 处理 [xxx]（问题）
```

**示例**：
- 使用 Rust 所有权系统处理内存安全问题
- 使用 Redis 缓存处理高并发查询性能问题
- 使用策略模式处理复杂业务逻辑的扩展性问题

**常见顾虑的解答**：

**顾虑1**："这个主题有人写过就不写了"

**解答**：
- 即使有人写过，你也可以写出自己的观点
- 你的具体实现和细节可能不同
- 你的经验和视角是独特的

**顾虑2**："我的想法不够有趣"

**解答**：
- 若你的读者还不了解这些结论，就不会感到无趣
- 有趣与否取决于读者当前的知识水平
- 对你来说可能已经习以为常的知识，对他人可能很有价值

---

### 第2步：确定读者对象

**目的**

确定写作口吻与内容深度。

**读者定位**

**建议**：面向**比自己稍弱一级**的读者

**为什么**：
- 确保常见概念有适当说明
- 不会假设过多前置知识
- 内容深度适中，既能满足进阶读者，也不会让初学者迷失

**通过两点精简内容**：

**1. 写作目标**
- 提供一套方法论
- 技术场景的解决方案
- 分享踩坑经验
- 其他：\_\_\_\_\_\_

**2. 读者对象**
- 来了解 xxx 领域相关知识的开发者
- 需要基于读者现有知识水平来写作

**示例**：
- 写作目标：提供使用 Rust 构建高性能服务的方法论
- 读者对象：有一定编程基础，但不熟悉 Rust 的后端开发者

---

### 第3步：结构设计

**技术文章固定模板（5部分）**：

这是一个经过验证的有效结构，适用于大多数技术文章：

#### 1. 背景 & 问题（10-15%）

**开头介绍**：
- 介绍背景，让读者了解文章定位
- 告诉读者能了解到什么
- 设置悬念或引发兴趣

**关键要素**：
- 问题场景描述
- 为什么这个问题重要
- 阅读本文能获得什么

**开头方式选择**：
- 从一个具体场景开始
- 从一个常见问题开始
- 从一次踩坑经历开始

#### 2. 方案或原理（25-30%）

**目标**：分点讲述核心概念

**关键要素**：
- 核心概念列表
- 技术方案原理
- 使用代码 + 图 + 列表辅助说明

**组织方式**：
- 按概念分类讲述
- 使用子标题和小标题
- 提供代码框架或架构图

#### 3. 实现步骤（30-35%）

**目标**：问题 + 方案的具体实现

**关键要素**：
- 分步骤说明
- 每个步骤有明确目标
- 完整可运行的代码
- 步骤之间有过渡

**写作建议**：
- 步骤编号清晰
- 每个步骤先说明目标
- 代码后紧跟关键说明
- 展示运行结果

#### 4. 示例 / 踩坑（15-20%）

**完整示例**：
- 提供完整的可运行代码
- 展示运行结果或输出

**常见错误**：
- 列出常见错误
- 分析错误原因
- 提供解决方案

**边界情况**：
- 说明需要注意的特殊场景
- 提供调试技巧

#### 5. 总结（5-10%）

**最后总结**：
- 快速总结文章内容
- 提出延伸思路
- 给予读者鼓励

**关键要素**：
- 核心要点回顾（3-5个）
- 适用场景说明
- 局限性诚实说明
- 延伸方向或资源推荐

使用 `assets/templates/blog-outline-template.md` 创建详细大纲。

---

### 第4步：编写初稿

**核心原则**

**不要追求完美初稿，只追求写完**

**推荐写作顺序**

按照这个顺序可以提高效率：

1. **先写第3部分（实现步骤）** - 这是你最熟悉的部分
2. **再写第2部分（方案或原理）** - 解释你刚才写的代码
3. **再写第4部分（示例/踩坑）** - 补充完整示例和经验
4. **再写第1部分（背景 & 问题）** - 现在你已经清楚要讲什么了
5. **最后写第5部分（总结）** - 总结你写的内容

**为什么这个顺序有效**：
- 从最熟悉的内容开始，降低心理阻力
- 写完核心内容后，引言和总结更容易把握
- 避免在开头纠结太久

**对照你的笔记**：
- 使用你的笔记作为内容来源
- 将笔记中的信息按5部分结构组织
- 补充笔记中缺失的过渡和解释

**初稿完成标准**：
- 5个部分都有内容
- 代码示例基本完整
- 逻辑基本通顺
- 不需要追求完美

---

### 第5步：优化

**核心检查问题（必查）**

发布前必须确保这三个问题的答案都是"是"：

#### 1. 是否跳过太多步骤？

**检查方法**：
- 每个步骤之间是否有清晰的逻辑连接
- 读者能否跟上思路而不需要"脑补"
- 是否有"显然"、"显而易见"等跳跃性表述

**改进方法**：
- 补充中间的逻辑推导
- 添加过渡句说明
- 提供更多上下文信息

#### 2. 是否假设了过多的前置知识？

**检查方法**：
- 专业术语是否有第一次出现时的解释
- 是否假设读者了解某些背景而没有说明
- 面向"比自己稍弱一级"的读者，内容是否合适

**改进方法**：
- 在第一次出现术语时添加解释
- 添加"前置知识"部分
- 用类比帮助理解复杂概念
- 链接到相关的基础资料

#### 3. 代码是否能运行？

**检查方法**：
- 所有代码是否在 IDE 中验证过
- 代码片段是否完整（非截断）
- 依赖和版本是否有说明

**改进方法**：
- 在 IDE 中完整运行一遍
- 补充缺失的导入和配置
- 说明依赖和版本要求
- 提供完整的示例代码

**其他优化检查**：

**可读性**：
- 段落长度适中（3-5句话）
- 使用列表呈现要点
- 关键内容有加粗或强调
- 复杂概念有类比或比喻

**格式**：
- Markdown 语法正确
- 代码块有语法高亮
- 链接全部有效
- 图片加载正常

**AI 辅助优化**：

推荐使用工具：
- **NotebookLM** - 获取文章优化建议
- 其他 AI 写作助手

**如何使用**：
1. 将初稿输入 AI 工具
2. 请求检查内容连贯性和可读性
3. 根据建议进行修改
4. 多次迭代优化

**获取反馈并迭代**：

**反馈渠道**：
- 同事或朋友审阅
- 技术社区预发布
- 收集初版读者的评论

**迭代方向**：
- 哪些部分最受欢迎
- 读者最常问的问题是什么
- 哪些说明还不够清楚

## 特殊场景处理

### 系列文章

**规划阶段**：
- 明确系列的主题和范围
- 规划每篇文章的边界
- 确保文章间的连贯性

**执行阶段**：
- 每篇文章都要有独立价值
- 添加系列索引或导航
- 在文末预告下一期

### 代码密集型文章

**组织策略**：
- 先展示完整代码
- 再分段解释关键部分
- 提供GitHub仓库链接

**注意事项**：
- 确保代码可运行
- 添加必要注释
- 说明依赖和环境

### 概念解释型文章

**写作技巧**：
- 从具体例子开始
- 使用类比和比喻
- 提供视觉辅助（图表）
- 渐进式深入

### 教程型文章

**结构建议**：
1. 明确的学习目标
2. 前置知识说明
3. 分步骤指导
4. 每步的验证检查
5. 常见问题排查

---

## 文章要有层次

### 开头介绍

**目的**：吸引读者，建立预期

**要素**：
- 介绍背景（让读者了解文章定位）
- 告诉读者能了解到什么
- 设置悬念等

### 中间细化

**目的**：按开头的介绍来细化内容

**技巧**：
- 加上结构标注和标题
- 方便读者跳转到感兴趣的模块
- 使用子标题组织内容

### 最后总结

**目的**：强化记忆，引导行动

**要素**：
- 快速总结文章内容
- 提出部分延伸思路
- 给予读者鼓励

---

## 添加点睛之笔：包装、发布和推广

### 包装优化

**标题优化**：
- 使用固定句式："使用 xxx 处理 xxx"
- 包含关键词
- 简洁有力

**描述撰写**：
- 一句话说明核心价值
- 包含主要关键词
- 控制在160字符以内

**标签选择**：
- 选择3-5个相关标签
- 使用平台的热门标签
- 确保标签准确反映内容

### 发布策略

**平台选择**：
- 个人博客
- Medium / Dev.to
- 掘金 / 知乎
- 公司技术博客

**发布时机**：
- 选择目标读者活跃的时间
- 避开节假日和重大事件

### 推广方法

**社交媒体**：
- Twitter / 微博分享
- 技术社区发布
- 相关群组分享

**互动引导**：
- 在文章结尾提出问题
- 鼓励读者分享经验
- 邀请反馈和评论

---

## 持续改进

### 数据分析

**关注指标**：
- 阅读量
- 评论数
- 点赞/收藏数
- 分享数

### 内容迭代

**定期回顾**：
- 哪些文章最受欢迎
- 读者最关心什么话题
- 哪些部分需要改进

**更新维护**：
- 更新过时的信息
- 补充新的示例
- 修正发现的错误
- 根据反馈改进表达

---

## 语言适配

### 中文技术博客特点

**语言风格**：
- 使用正式但不过于学术的语言
- 技术术语保持英文原文
- 避免过度使用网络用语

**格式习惯**：
- 中英文之间添加空格
- 使用中文标点符号
- 代码注释使用中文

### 英文技术博客特点

**语言风格**：
- 直接简洁的表达
- 主动语态优先
- 避免过度修饰

**格式习惯**：
- 正确的英文空格和标点
- 技术术语首字母大写规则
- 缩写的正确使用

---

## 总结

费曼学习法的核心是"以教代学"。通过写作技术博客，你不仅能帮助他人学习，也能检验和深化自己的理解。

记住这个流程：

**基于笔记创作**：
1. 准备笔记文件
2. 分析笔记内容
3. 映射到5部分结构
4. 转化和补充
5. 优化验证

**自主创作**：
1. 从日常开发中确定主题
2. 面向比自己稍弱一级的读者
3. 使用5部分固定结构
4. 先追求写完，再追求完美
5. 检查三个核心问题

最重要的是：开始写，完成它，发布出去。

**笔记是给未来的自己看的，文章是给别人看的。**
