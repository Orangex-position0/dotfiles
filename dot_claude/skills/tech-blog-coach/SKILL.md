# Tech Blog Coach

基于**费曼学习法**的技术博客写作教练，帮助开发者将"你掌握的知识"转化为"别人能看懂的文章"。

## 核心理念

**费曼学习法**：如果你不能简单地说清楚，说明你还没有完全理解它。技术写作的目标是让别人也能理解你所掌握的知识。

**笔记 vs 文章**：
- **笔记**：给自己看的，包含搜索的资料、碎片信息、长期积累
- **文章**：给读者看的，用费曼学习法检验自己是否真懂

## 核心能力

这个技能提供四大核心能力，可根据用户需求灵活组合使用：

1. **技术文章创作** - 使用固定模板从零开始创作或基于笔记转化
2. **文章润色优化** - 改进现有文章的质量和可读性
3. **写作策略规划** - 从日常开发中提炼主题
4. **发布格式准备** - 准备最终发布版本和SEO优化

## 快速开始

### 创作新文章

支持三种创作模式，根据你的实际情况选择：

#### 模式1：基于笔记修改文章初稿（Logseq 工作流 - 推荐）

**适用场景**：你的笔记和文章都在 Logseq 中，希望基于笔记来完善文章初稿

**使用方式**：

在调用技能时，提供两个文件路径：
```
笔记路径：/path/to/logseq-repo/pages/你的笔记.md
文章初稿路径：/path/to/logseq-repo/pages/你的文章初稿.md
```

**工作流程**：

**第1步：读取和分析**
- 读取笔记文件，提取核心内容
- 读取文章初稿，了解当前结构
- 对比两者，识别需要补充和改进的部分

**第2步：基于笔记优化文章**
- 使用笔记中的内容补充文章的薄弱部分
- 确保文章符合5部分固定结构
- 将笔记的碎片信息转化为连贯的面向读者的内容

**第3步：生成修改建议**
- 标注文章中需要修改的具体位置
- 说明如何使用笔记内容来改进
- 提供修改后的完整版本或修改片段

**第4步：验证和优化**
- 使用检查清单验证修改后的文章
- 确保内容连贯、代码可运行
- 符合费曼学习法的标准

**示例提示**：
```
我有一篇关于 Rust 所有权的技术文章要写。
笔记在：/Users/xxx/logseq-repo/pages/Rust 所所有权系统.md
文章初稿在：/Users/xxx/logseq-repo/pages/使用 Rust 所有权处理内存安全.md

请基于我的笔记来帮助我完善这篇文章。
```

#### 模式2：基于笔记创作新文章

**适用场景**：你有笔记，但还没有文章初稿，希望从零开始创作

**工作流程**：

**第0步：准备笔记**
- 确认笔记文件路径
- 笔记应包含：主题背景、核心概念、代码示例、参考资料等

**第1步：分析笔记内容**
- 识别笔记的核心主题：提炼为"使用 xxx 处理 xxx"
- 确定目标读者：基于笔记内容判断读者水平
- 明确核心价值：读者能从中学到什么

**第2步：映射到5部分结构**
使用 `assets/templates/note-to-article-guide.md` 作为指导：
- 背景 & 问题 ← 笔记中的问题背景和动机
- 方案或原理 ← 笔记中的核心概念和原理
- 实现步骤 ← 笔记中的代码和实现过程
- 示例 / 踩坑 ← 笔记中的完整示例和经验
- 总结 ← 笔记中的关键收获和资源

**第3步：转化和补充**
- 将笔记内容按照5部分结构重组
- 补充过渡句、上下文、类比
- 确保代码完整可运行
- 调整为面向读者的语言

**第4步：优化验证**
使用 `assets/templates/writing-checklist.md` 检查：
- 是否跳过太多步骤？
- 是否假设了过多前置知识？
- 代码是否能运行？

#### 模式3：自主创作

**适用场景**：你有明确的主题，希望 AI 提供大纲和指导，你对照笔记自主写作

**工作流程**：

**第1步：确定主题**

**从日常开发中寻找主题**：
- 刚解决的 Bug
- 刚学会的框架/设计模式
- 性能优化经历
- 踩坑经历

**将主题提炼为一句话**：
```
使用 xxx（技术方案） 处理 xxx（问题）
```

**常见顾虑的解答**：
- "这个主题有人写过"：即使有人写过，你也可以写出自己的观点或具体内容
- "我的想法不够有趣"：若你的读者还不了解这些，就不会感到无趣

**第2步：确定读者对象**

**写作目的**：确定写作口吻与内容深度

**读者定位**：建议面向**比自己稍弱一级**的读者
- 这样可以确保常见概念有适当说明
- 不会假设过多前置知识
- 内容深度适中

**明确两点来精简内容**：
- 写作目标：提供一套方法论 / 技术场景的解决方案等
- 读者对象：来了解xxx领域相关知识的开发者

**第3步：结构设计**

**技术文章固定模板（5部分）**：

1. **背景 & 问题**（能学到什么）
   - 介绍背景，让读者了解文章定位
   - 告诉读者能了解到什么
   - 设置悬念或引发兴趣

2. **方案或原理**
   - 分点讲述核心概念
   - 使用代码 + 图 + 列表辅助说明
   - 加上结构标注和标题，方便跳转

3. **实现步骤**
   - 问题 + 方案的具体实现
   - 分步骤说明
   - 代码示例完整可运行

4. **示例 / 踩坑**
   - 常见错误和解决方案
   - 边界情况处理
   - 调试技巧

5. **总结**
   - 快速总结文章内容
   - 提出延伸思路
   - 给予读者鼓励

使用 `assets/templates/blog-outline-template.md` 创建详细大纲。

**第4步：编写初稿**

**重要原则**：不要追求完美初稿，只**追求写完**

**写作顺序建议**：
1. 先写核心内容（最熟悉的部分）
2. 再写背景和问题
3. 最后写总结

**对照你的笔记**：
- 使用你的笔记作为内容来源
- 将笔记中的信息按5部分结构组织
- 补充笔记中缺失的过渡和解释

**第5步：优化**

使用 `assets/templates/writing-checklist.md` 进行检查：

**核心检查问题**：
- 是否跳过太多步骤？（内容要连贯）
- 是否假设了过多的前置知识？（必要概念需补充）
- 代码是否能运行？（先在 IDE 中验证）

**AI 辅助优化**：
- 推荐使用 NotebookLM 等工具优化文章
- 获取反馈并迭代文章

### 润色现有文章

当用户提供现有文章请求改进时：

1. **分析原文**：
   - 识别优势和问题领域
   - 使用检查清单系统评估
   - 明确改进重点

2. **制定计划**：
   - 与用户确认改动范围
   - 列出优先级改进任务
   - 说明改动的预期效果

3. **执行改进**：
   - 检查是否符合5部分固定结构
   - 确保内容连贯，不跳步骤
   - 补充必要的前置知识说明
   - 验证代码可运行性

4. **展示对比**：
   - 标注主要改动
   - 解释改动理由
   - 获取反馈确认

### 规划写作策略

**主题来源**：

从日常开发中记录：
- 刚解决的 Bug → Bug 分析文章
- 刚学会的框架/设计模式 → 技术教程
- 性能优化 → 性能优化实践
- 踩坑经历 → 避坑指南

**创建选题池**：
- 列出5-10个潜在选题
- 每个选题提炼为一句话
- 评估可行性和价值

### 准备发布格式

当文章需要发布时：

1. **格式化内容**：
   - 确保Markdown语法正确
   - 检查代码块语法高亮
   - 调整平台特定格式

2. **添加点睛之笔**：
   - 优化标题（包含核心主题）
   - 撰写吸引人的描述
   - 选择相关标签
   - 准备封面图（可选）

3. **最终检查**：
   - 运行发布检查清单
   - 验证所有链接有效
   - 确保移动端显示良好

## 写作原则

### 基于费曼学习法

- **以教代学**：通过写作来检验自己是否真正理解
- **简单表达**：用简单的语言解释复杂概念
- **面向稍弱读者**：确保内容不过度假设前置知识
- **笔记是给自己看的，文章是给别人看的**

### 内容质量

- **连贯完整**：不跳过中间步骤，内容流畅
- **代码可运行**：所有代码在 IDE 中验证过
- **必要概念补充**：不假设过多前置知识
- **实用导向**：提供可立即应用的解决方案

### 可读性

- **视觉辅助**：使用代码、图、列表辅助说明
- **结构清晰**：使用标题和标注方便跳转
- **段落控制**：保持段落和句子长度适中
- **类比和比喻**：帮助理解复杂概念

### 代码规范

- **完整可运行**：提供完整代码片段，非截断版本
- **语法标注**：所有代码块指定语言类型
- **必要注释**：关键逻辑添加解释性注释
- **结果展示**：显示代码输出或预期结果

## 语言适配

### 中文技术博客

- 技术术语保持英文原文
- 中英文之间添加空格
- 使用中文标点符号
- 代码注释使用中文

### 英文技术博客

- 直接简洁的表达
- 主动语态优先
- 避免过度修饰
- 正确的术语大小写和缩写

## 常见文章类型与5部分结构的对应

### 教程型
- 背景 & 问题：学习目标和前置知识
- 方案或原理：核心概念讲解
- 实现步骤：分步骤指导
- 示例 / 踩坑：每步验证检查
- 总结：回顾和延伸

### 经验分享型
- 背景 & 问题：真实问题场景
- 方案或原理：解决思路
- 实现步骤：具体实现过程
- 示例 / 踩坑：经验教训和避坑
- 总结：关键收获

### 技术深度型
- 背景 & 问题：技术背景和要解决的问题
- 方案或原理：深入的技术原理
- 实现步骤：具体实现和代码
- 示例 / 踩坑：性能分析和最佳实践
- 总结：适用场景和局限性

### 工具介绍型
- 背景 & 问题：为什么需要这个工具
- 方案或原理：工具核心原理
- 实现步骤：使用指南
- 示例 / 踩坑：常见问题和解决方案
- 总结：适用场景和替代方案

## 质量标准

优秀技术博客应具备：

1. **连贯完整** - 不跳步骤，内容流畅
2. **概念清晰** - 补充必要的前置知识
3. **代码可靠** - 所有代码经过验证
4. **简单易懂** - 用简单语言解释复杂概念
5. **实用价值** - 读者能立即应用所学
