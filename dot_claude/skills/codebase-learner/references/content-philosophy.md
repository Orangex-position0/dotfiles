# Content Philosophy

> **When to read this:** During Phase 1（生成项目快照）和 Phase 2（生成学习单元）。这些原则指导所有内容决策——展示什么、如何解释、如何验证理解。

## 信息密度优先

读者是有 CS 背景的开发者，不需要把概念嚼碎了喂。内容应该像一份好的技术博客或 RFC，而不是入门教程。

**核心原则：**
- 每个段落都应有信息增量。如果删掉一段不影响理解，就不该写
- 代码片段配合设计意图解释，不需要逐行翻译
- 图示用于表达关系和流程，不是装饰
- 大胆使用技术术语，项目特有的概念通过注释或术语表解释

**文本限制：**
- 一个观点 3-5 句话讲清楚，不超过一个短段落
- 超过 5 个要点的列表 → 转为卡片或表格
- 超过 3 步的流程 → 转为步骤卡片或流程图

**何时使用可视化：**
- 模块间依赖关系 → 架构图或依赖图
- 请求处理链路 → 数据流动画或步骤卡片
- 多个组件协作 → 时序图或调用链追踪
- 方案对比 → 并排对比视图
- 目录结构 → 文件树（带职责标注）

---

## 代码 ↔ 设计意图注释

代码片段不做逐行翻译。左侧展示真实代码，右侧解释**设计意图**——为什么这样写、解决了什么问题、有哪些替代方案。

**示例：**

```
左侧代码：
const cache = new LRUCache({ max: 100, ttl: 60000 });

右侧意图注释：
项目使用 LRU 缓存策略（最近最少使用淘汰），最多缓存 100 条结果，
60 秒后过期。这是一个典型的空间换时间优化——在 API 调用密集场景下
减少重复请求。替代方案：考虑 Redis 做分布式缓存以支持多实例部署。
```

**关键规则：**
- 代码必须是原始代码的精确复制，不做简化或修改
- 选择能体现设计模式或关键决策的片段（通常 5-15 行）
- 右侧解释聚焦于：设计意图、解决的问题、权衡取舍、替代方案
- 不解释语法细节（读者是开发者，知道 `const` 是什么）

---

## 上下文优先于结论

不仅要告诉读者"这个项目用了 X 模式"，更要解释：

1. **为什么选 X 而不是 Y**——设计决策背后的约束条件
2. **X 的代价是什么**——没有银弹，每个选择都有 trade-off
3. **在什么条件下 X 会失效**——适用边界

这是区分"看懂代码"和"理解设计"的关键。一个只说了"项目用了 Repository 模式"的学习材料是失败的；一个说了"项目用了 Repository 模式，因为需要同时支持 MySQL 和 MongoDB，如果只用一种数据库可以直接用 DAO 模式简化代码"的才是有价值的。

## 渐进式深入原则

每个学习单元应遵循这个结构：

1. **What**：这个模块/层/组件是做什么的（1-2 句）
2. **Why**：为什么需要它，解决了什么问题（2-3 句，带上下文）
3. **How**：它是怎么实现的（代码片段 + 设计意图注释）
4. **Trade-offs**：这种设计的优劣和适用场景
5. **Connection**：它和项目的其他部分如何关联

不是每个部分都要大段文字——有些用一个图表、一个代码片段就能说清楚。但这个顺序不能省略。

## 术语与概念注释

读者有 CS 基础，不需要解释"什么是函数"、"什么是 API"。需要注释的是：

- **项目特有的概念**：领域模型中的业务术语、自定义框架的概念
- **非通用技术**：特定框架的专有 API（如 Spring 的 `@Transactional`、MyBatis 的动态 SQL）
- **设计模式的特定应用**：不是解释"什么是策略模式"，而是解释"这里为什么用策略模式、它在这个场景下怎么工作"

**注释方式（HTML 输出）：**
- 使用 `data-definition` 的 term 标签，但只在项目特有概念上使用
- 定义应包含：这个概念在这里的含义 + 为什么项目选择用它

**注释方式（Markdown 输出）：**
- 首次出现时用 `[术语](* glossary.md 中的锚点)` 格式
- 或在章节末尾添加术语表块

## 测验设计：测试架构判断力

测验的目标是验证读者是否理解了设计决策背后的 reasoning，而不是记忆代码细节。

**好的测验问题：**

1. **设计决策题**："项目在处理并发时选择了乐观锁而不是悲观锁，如果在 X 场景下换成悲观锁，会有什么影响？"——测试对权衡的理解
2. **架构判断题**："如果要给这个项目添加 X 功能，应该改哪些模块？为什么？"——测试对架构的理解
3. **代码追踪题**："一个请求从入口到数据库返回，经过了哪些层？每一层做了什么？"——测试对流程的理解
4. **方案对比题**："项目用消息队列解耦了模块 A 和 B，如果改成同步调用，在什么场景下更好？"——测试对 trade-off 的理解

**不考的：**
- 代码语法细节
- 文件名记忆
- 框架 API 签名
- 可以通过搜索找到的事实

**答案解释**：每个选项都应该解释为什么对/为什么错，让读者即使选错也能学到东西。

---

## 隐喻使用原则

不再强制每个模块都有隐喻。隐喻只在以下情况使用：

- 概念本身很抽象，需要具象化辅助理解（如分布式一致性）
- 项目用了独特的类比来解释自己的架构（直接引用项目的类比）
- 读者可能没有接触过的非主流技术领域

对于后端开发者熟悉的概念（MVC、依赖注入、缓存、ORM 等），直接使用技术术语，不需要绕弯子。
